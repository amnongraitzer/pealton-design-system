<objective>
Implement PLT-081 (issue #9): the Pealton Design System inline-edit module — a
framework-free, zero-runtime-dep, STANDALONE per-field editor factory
`createFieldEditor(mount, options)` that carries the ONE edit grammar the design
system defines (docs/TABLE-CONTRACT.md §Inline edit, §Concurrency, §Confirm
ladder rung 1): hover→in-place edit with explicit ✓ confirm / ✕ cancel,
Enter=confirm / Esc=cancel, a wait-not-optimistic busy state, text/select/number
controls with a per-field validation hook, and a 409-conflict path that refreshes
to the server value with an inline notice and never silently overwrites.

The factory is REQUIRED to be consumable OUTSIDE a table cell — Epic 4's detail
drawer (a separate task) reuses the exact same grammar. This task also wires it
into `pds-table` cells so a table column edits with the identical grammar, adds
an editable-column demo to `docs/table.html` driven by a fake async saver
(success / failure / conflict), and ships pytest tests.

Purpose: one edit grammar, defined once, used in table rows AND the drawer — no
per-surface re-implementation. Output: `pds-field-editor.js` (new), a thin
editable-column seam in `pds-table.js`, appended `tabulator.css`, the demo, and
`tests/test_field_editor.py` (+ a node/DOM-stub harness).

Scope guards (do NOT build): any HR endpoint wiring (PLT-084), bulk editing,
the destructive confirm ladder rungs 2/3 (dialog + typed-count — those are bulk
/ destructive actions, out of scope here), replacing Tabulator, real-estate
tables. The module is app-agnostic: NO candidate/HR field names, NO endpoints,
NO Hebrew business strings baked in. The stage optimistic-sync exception in the
contract is a consumer choice (the consumer's onSave decides) — the DS default
is wait-not-optimistic.
</objective>

  <name>Task 1: Standalone factory — createFieldEditor (grammar, types, validate, busy, cancel/confirm, Esc no-bubble)</name>
  <files>pealton_design_system/static/pds-field-editor.js</files>
  <action>
Create a NEW vanilla-JS file as an IIFE opening with "use strict", matching the
header-comment style of pds.js / pds-table.js. ZERO runtime deps. Expose the API
on a `root` that is `window` in the browser and `globalThis` under Node (so the
test harness can load it): `root.createFieldEditor = createFieldEditor`,
`root.PDS = root.PDS || {}`, `root.PDS.createFieldEditor = createFieldEditor`,
and additionally `if (typeof module !== "undefined" && module.exports)
module.exports = { createFieldEditor }` for the node harness. No behaviour change
in the browser.

Define an internal `escapeHtml(value)` helper (same five-metachar escape as
pds-table.js) — every injected string routes through it.

Define `createFieldEditor(mount, options)`:
  - `mount`: a DOM node or selector string the editor renders into.
  - options: `type` ("text"|"select"|"number", default "text"); `value` (initial
    committed value); `options`/`choices` for select (array of {value,label} or
    bare values); `validate(value)` optional hook (return true/undefined =
    valid, or a string/false = invalid → inline validation message, stays open);
    `onSave(value)` REQUIRED promise-returning commit callback; `onCommit(value)`
    optional (after a successful save); `onConflict(current)` optional (after a
    409-shaped rejection); `onCancel()` optional; `conflictText` / `errorText`
    optional per-view copy (escaped; neutral non-business fallback — NOT Hebrew).
  - Render a display cell showing the current value plus a hover/focus edit
    affordance. `open()` swaps in the control (text input / number input /
    select built from choices), pre-filled and focused, with ✓ and ✕ buttons.
  - `confirm()` (✓ or Enter): if busy → no-op (blocks double-submit). Run
    validate; on invalid show the escaped message inline and stay open. If the
    value is unchanged, close without calling onSave (no-op commit). Otherwise
    enter BUSY: disable the control + both buttons, show an in-control spinner,
    and await `onSave(value)`. On resolve → commit (store new value, re-render
    display, call onCommit, close). On reject → if the error looks 409-shaped
    (`err && err.conflict === true`) run the CONFLICT path; else clear busy,
    show an escaped inline error, keep the editor open and re-enabled for retry.
  - CONFLICT path: read `err.current` as the fresh server value; store it as the
    new committed value; re-render the display showing the fresh value; show an
    inline notice (conflictText, escaped) telling the user to re-apply; call
    `onConflict(current)`; return to a re-openable display state — NEVER silently
    overwrite the server value with the stale edit.
  - `cancel()` (✕ or Esc): restore the prior committed value, close, call
    onCancel. It must NOT commit.
  - Keydown handling on the control: Enter → confirm; Escape → cancel AND call
    `event.stopPropagation()` (and preventDefault) so Esc does NOT bubble to an
    outer native <dialog> and close it (contract Keyboard: "Esc rules apply
    uniformly"; a drawer-hosted editor must swallow its own Esc).
  - Return a controller: `open`, `close`, `confirm`, `cancel`, `getValue`,
    `isBusy`, `isEditing`, `destroy`, and `element` (the mount).

Document each block with a short purpose comment. No business field names, no
external URL, no clever tricks (CLAUDE.md).
  </action>
